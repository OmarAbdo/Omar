## Most Technical Blog Posts Are Documentation Wearing a Costume

Here's the pattern: a developer learns something new, opens a markdown file, and writes "How to Set Up X with Y." What follows is a step-by-step walkthrough that duplicates the official docs, adds no opinion, provides no context for *why* you'd choose this approach over alternatives, and could have been generated by a language model.

Nobody finishes reading these. They skim to the code block they need, copy-paste it, and leave. The article gets traffic from Google but no engagement, no shares, no readers who come back for the next one.

I've been writing technical articles for my portfolio for a while now, and the ones that actually get read — the ones people message me about — share a common trait. They don't explain *what*. They explain *why*. The "what" is documentation. The "why" is insight. Documentation is free and abundant. Insight is rare and valuable.

## The Opening Paragraph Is 80% of Success

Readers decide whether to commit to your article within the first three sentences. Not the first three paragraphs — the first three *sentences*. This means your opening needs to do one of three things:

1. **Challenge something the reader believes.** "Goroutines are not threads" immediately creates tension. If the reader thought they were threads, they need to keep reading to resolve the dissonance.

2. **Promise a specific outcome.** "After reading this, you'll understand why your Docker container isn't a VM" gives the reader a concrete reason to invest 10 minutes.

3. **Describe a problem the reader has experienced.** "You've probably run `npm install`, watched 400 packages download, and wondered how your package manager figured out the right order." The reader nods and thinks, "yeah, I have."

What doesn't work: "In this article, I will discuss..." — nobody cares what you'll discuss. They care what they'll *gain*. "X is a popular technology used by many companies" — this is filler, and your reader's back button is one click away.

The opening of every article I write now follows the same structure: make a claim, establish stakes, preview the payoff. Three sentences. If I can't do it in three sentences, I don't understand my own article well enough.

## Structure That Works: Claim, Evidence, Nuance, Takeaway

The worst technical articles are streams of consciousness. The second worst are rigid "Introduction, Background, Method, Results" academic papers.

What works for technical articles is a pattern I think of as **claim-evidence-nuance-takeaway**, repeated for each section:

1. **Claim:** State something opinionated. "Adjacency lists beat adjacency matrices for 95% of real-world graphs."
2. **Evidence:** Show why. Code, benchmarks, reasoning. "An adjacency matrix for a million-node social network requires a trillion entries. An adjacency list stores only existing connections."
3. **Nuance:** Acknowledge the trade-off. "But if you need O(1) edge lookups on a dense graph, the matrix wins."
4. **Takeaway:** What should the reader do? "Default to adjacency lists. Reach for a matrix only when you've measured that edge lookups are your bottleneck."

This structure works because it respects the reader's intelligence while being honest about complexity. You're not writing a sales pitch (all claim, no nuance). You're not writing a textbook (all evidence, no opinion). You're writing something useful — an experienced person's considered take.

Each section header should advance the story. "BFS and DFS: The Two Ways to Walk a Graph" is better than "Traversal Algorithms." The first tells the reader what they're about to learn and why it's framed as a choice. The second is a label that could be a chapter heading in any textbook.

## Code Samples: Short Enough to Read, Real Enough to Use

Code samples are where most technical articles either succeed or fail. Here are the mistakes I see constantly:

**Too long.** If your code sample is 60 lines, the reader's eyes glaze over. They won't trace through the logic. They'll skip it entirely, which means your "evidence" for the claim above is invisible.

**Too toy.** `function add(a, b) { return a + b; }` teaches nothing. The reader knows that functions take arguments and return values. Show them something they'd actually encounter — real error handling, real data shapes, real constraints.

**No context.** A code block dropped into the article without explanation above and below is a puzzle, not an example. Say what the code does before showing it. Say what's interesting about it after.

Here's what I aim for: 5-25 lines per code sample. Enough to show a real pattern, short enough to read in one pass. If the code needs more context, add comments inline — but only for the non-obvious parts.

```typescript
// Bad: too toy to be useful
function getUser(id: string) {
  return db.find(id);
}

// Better: shows a real pattern with error handling and types
async function getUser(id: string): Promise<User | null> {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);

  const user = await db.users.findUnique({ where: { id } });
  if (user) await redis.set(`user:${id}`, JSON.stringify(user), 'EX', 300);
  return user;
}
```

The second example teaches cache-aside pattern, TTL-based expiration, and nullable return types in 7 lines. That's the density you're aiming for — every line carries weight.

One more thing: make sure your code actually runs. I've read articles where the code samples have syntax errors, wrong types, or APIs that don't exist. Nothing destroys credibility faster. If you're showing TypeScript, run `tsc` on it. If you're showing a shell command, run it first.

## Voice: Opinionated but Honest

The technical articles that stick with me have a clear voice. The author has opinions, defends them, and acknowledges when they might be wrong.

Compare:

> "There are many state management solutions available for React applications. Redux, MobX, Zustand, and Jotai are popular choices. Each has advantages and disadvantages."

With:

> "Most React apps don't need a state management library. useState and useContext cover 90% of use cases. The other 10%? I'd reach for Zustand before Redux every time — less boilerplate, simpler mental model, and the Redux DevTools aren't worth the ceremony."

The first is Wikipedia. The second is a person with experience sharing what they've learned. Readers follow people, not information aggregators. If they wanted a neutral comparison, they'd read the docs.

But — and this is crucial — opinionated doesn't mean dismissive. "Redux is garbage" is not an opinion, it's laziness. "Redux adds ceremony that most applications don't need, but for apps with complex state machines and time-travel debugging requirements, the structure pays for itself" is an opinion with nuance. The reader trusts you because you've shown you understand the other side.

Every strong claim in your article should have a caveat somewhere nearby. Not to hedge, but to demonstrate that you've actually thought about the trade-offs. The reader is smart. Treat them that way.

## SEO Doesn't Matter If Nobody Finishes Reading

I've watched developers obsess over keywords, meta descriptions, and header tag hierarchies while writing articles that nobody would want to read for more than 30 seconds.

SEO gets people to your article. The writing determines whether they stay. A well-written article with mediocre SEO will build a loyal readership through shares and word of mouth. A poorly written article with perfect SEO will get clicks and immediate bounces, which eventually tanks your SEO anyway.

The things that actually matter for technical article reach:

1. **A title that creates curiosity or promises value.** "Graph Theory: The Human Explanation" works better than "Introduction to Graph Theory" because it implies a perspective, not just information.
2. **A description that tells the reader what they'll gain.** Not what the article covers — what *they* get from reading it.
3. **Sharing the article where your audience already is.** Dev.to, Hacker News, relevant Discord servers, Twitter/X. One share in the right community outperforms months of SEO optimization.
4. **Writing consistently.** One great article a month beats four mediocre articles a week. Readers subscribe to quality, not volume.

## The Meta-Lesson

This article is, obviously, self-referential. I'm using the same structure I'm describing: opinionated opening (most blog posts are documentation in disguise), evidence (specific examples of what works and doesn't), nuance (acknowledge trade-offs, respect the reader), practical takeaways (code sample guidelines, structural patterns).

I'm drawing from my own experience writing the articles on this portfolio site. Some of them worked — they got messages, sparked conversations, taught something useful. Some of them didn't — they were too academic, too generic, or too long.

The articles that worked had three properties:

1. **They had a point of view.** Not "here's how X works" but "here's why X matters and when you should use it."
2. **They were honest about trade-offs.** The reader finished trusting the author's judgment because the author showed their reasoning, including the parts where the answer is "it depends."
3. **They respected the reader's time.** Every section justified its existence. If a section could be cut without losing the core argument, it got cut.

Technical writing is not separate from technical skill. The ability to explain something clearly is proof that you understand it. The muddled explanation reveals muddled thinking. If you can't write a clear article about your architectural decision, there's a chance the decision itself isn't clear.

Write the article you wish existed when you were learning. That's the bar, and it's the only one that matters.
